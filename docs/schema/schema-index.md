# AgentHub 数据库设计规范

## 1. 概述

本文件定义了 AgentHub 项目的数据库设计规范和 Schema 管理流程，确保数据库结构的一致性、可维护性和性能。

## 2. 设计原则

### 2.1 数据建模原则
- **单一职责原则**: 每个表应只包含与一个主题相关的数据
- **数据完整性原则**: 确保数据的准确性和一致性
- **性能优化原则**: 考虑查询性能，合理设计索引
- **可扩展性原则**: 设计应支持未来的扩展和变更
- **安全性原则**: 保护敏感数据，实现适当的访问控制

### 2.2 命名规范

#### 2.2.1 表名
- 使用 snake_case（下划线分隔）命名表
- 表名应使用复数形式
- 避免使用缩写，除非是广泛接受的缩写
- 示例：`users`, `agents`, `transactions`

#### 2.2.2 字段名
- 使用 snake_case（下划线分隔）命名字段
- 字段名应使用单数形式
- 使用有意义的名称，避免缩写
- 示例：`user_id`, `agent_name`, `created_at`

#### 2.2.3 索引名
- 使用 `idx_` 作为前缀
- 包含表名和字段名
- 示例：`idx_users_email`, `idx_agents_category_id`

#### 2.2.4 关系名
- 使用描述性名称表示表之间的关系
- 遵循 Prisma 的关系命名约定
- 示例：`user` (一对一), `agents` (一对多), `users` (多对多)

### 2.3 数据类型选择

- **字符串**: 使用 `String` 类型，合理设置长度限制
- **整数**: 使用 `Int` 或 `BigInt` 类型，根据数据范围选择
- **浮点数**: 使用 `Float` 或 `Decimal` 类型，根据精度要求选择
- **布尔值**: 使用 `Boolean` 类型
- **日期时间**: 使用 `DateTime` 类型
- **JSON**: 仅在必要时使用 `Json` 类型，优先考虑关系模型

### 2.4 默认值和约束

- **必填字段**: 明确标记 `@unique` 和 `@default` 约束
- **默认值**: 为适当的字段设置合理的默认值
- **唯一约束**: 为需要唯一标识的字段添加 `@unique` 约束
- **外键约束**: 正确定义表之间的关系

### 2.5 注释规范

- 为表和字段添加清晰的注释
- 注释应说明表或字段的用途
- 使用 Prisma 的注释语法

**示例：**
```prisma
model User {
  id        String   @id @default(cuid())
  email     String   @unique @db.VarChar(255) /// 用户邮箱，用于登录和通信
  name      String?  @db.VarChar(100)        /// 用户名称
  created_at DateTime @default(now())        /// 用户创建时间
  updated_at DateTime @updatedAt             /// 用户信息更新时间
}
```

## 3. Schema 管理

### 3.1 管理职责

Prisma Schema 由项目负责人统一管理，所有 Schema 变更必须经过项目负责人审核和批准。

### 3.2 变更流程

#### 3.2.1 提交需求
- 在 PR 描述中详细列出需求
- 包括：表名、字段名、类型、默认值等信息
- 说明变更的原因和影响

#### 3.2.2 负责人审核与修改
- 项目负责人审核需求的合理性和完整性
- 根据需求修改 `prisma/schema.prisma` 文件
- 确保变更符合数据库设计最佳实践

#### 3.2.3 合入与同步
- 项目负责人将修改合入 main 分支
- 各开发分支通过 rebase 获取最新的 schema 变更
- 运行 `npx prisma generate` 更新客户端代码
- 运行 `npx prisma migrate dev` 应用数据库迁移

### 3.3 禁止操作

- 开发人员不得直接修改 `prisma/schema.prisma` 文件
- 不得绕过项目负责人进行 Schema 变更
- 不得在未更新 Schema 的情况下修改相关代码

### 3.4 版本控制

- Schema 变更必须在 Git 中进行版本控制
- 每个 Schema 变更必须有对应的 commit 记录
- Commit 信息应清晰描述变更内容

## 4. 最佳实践

### 4.1 设计最佳实践

- **保持 Schema 的简洁性和一致性**
- **使用有意义的表名和字段名**
- **为必填字段设置合理的默认值**
- **添加适当的注释说明字段用途**
- **考虑性能优化（如索引）**
- **确保数据类型的正确性和完整性**
- **避免过度规范化**
- **使用连接表管理多对多关系**

### 4.2 查询性能优化

- **添加适当的索引**: 为频繁查询的字段添加索引
- **避免全表扫描**: 确保查询使用索引
- **合理使用关联查询**: 避免过多的嵌套关联查询
- **使用分页**: 对大型结果集使用分页

### 4.3 安全性考虑

- **加密敏感数据**: 对密码等敏感数据进行加密存储
- **限制字段访问**: 只返回必要的字段
- **防止 SQL 注入**: 使用 Prisma 的参数化查询
