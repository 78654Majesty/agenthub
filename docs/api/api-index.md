# AgentHub API 文档规范

## 1. 概述

本文件定义了 AgentHub 项目前后端交互接口文档的编写规范、结构和格式，确保接口文档具有一致性、完整性和可理解性，便于前后端开发人员的协作和集成。

## 2. 文档目的

- 明确前后端交互的接口定义和规范
- 提供接口的详细说明和使用示例
- 建立接口文档的版本控制和更新机制
- 确保接口文档与实际实现的一致性

## 3. API 文档结构

每个 API 文档应包含以下核心部分：

### 3.1 文档信息
- **文档标题**: 清晰描述 API 的范围和内容
- **文档版本**: 版本号，遵循语义化版本规范
- **文档状态**: 草稿/评审中/已批准/已实施/已完成
- **创建日期**: 文档首次创建的日期
- **更新日期**: 文档最近一次更新的日期
- **创建人**: 文档的主要创建者
- **审核人**: 文档的审核人员

### 3.2 API 概述
- **API 名称**: API 的名称
- **API 描述**: API 的功能和用途
- **API 基础 URL**: API 的基础 URL
- **认证方式**: API 的认证方式（如 JWT、API Key 等）
- **请求格式**: API 请求的数据格式（如 JSON、FormData 等）
- **响应格式**: API 响应的数据格式（如 JSON 等）
- **错误处理**: API 的错误处理机制和错误码

### 3.3 接口列表
- 提供 API 接口的列表，包含接口名称、路径、方法和简要描述
- 以表格形式展示，便于快速查阅

### 3.4 接口详情

每个接口的详情应包含：
- **接口路径**: API 接口的完整路径
- **请求方法**: HTTP 请求方法（如 GET、POST、PUT、DELETE 等）
- **接口描述**: 接口的详细描述
- **认证要求**: 是否需要认证
- **请求参数**: 
  - **路径参数**: 路径中的参数
  - **查询参数**: URL 查询参数
  - **请求体**: 请求体的结构和示例
- **响应数据**: 
  - **成功响应**: 成功响应的结构和示例
  - **错误响应**: 错误响应的结构和示例
- **状态码**: 接口可能返回的 HTTP 状态码
- **使用示例**: 接口的使用示例代码（如 curl、JavaScript、Python 等）

### 3.5 数据模型
- 定义 API 使用的数据模型
- 包含模型的字段、类型、描述和约束条件
- 提供示例数据

### 3.6 认证与授权
- 详细说明 API 的认证机制
- 提供认证流程和示例
- 说明不同接口的授权要求

### 3.7 速率限制
- 说明 API 的速率限制策略
- 提供速率限制的规则和示例

### 3.8 变更日志
- 记录 API 的变更历史
- 包含版本、变更内容、变更日期和变更人

## 4. API 文档类型

根据 API 的范围和内容，API 文档可分为以下几类：

| 分类 | 描述 | 示例 |
|------|------|------|
| 全局 API 文档 | 整个系统的 API 文档 | AgentHub API 文档 |
| 模块 API 文档 | 某个模块的 API 文档 | Gateway API 文档 |
| 版本 API 文档 | 某个版本的 API 文档 | AgentHub API v1.0 文档 |
| 接口组 API 文档 | 某个接口组的 API 文档 | 支付接口 API 文档 |

## 5. API 文档状态

API 文档的生命周期包含以下状态：

- **草稿**: 文档正在编写中，尚未完成
- **评审中**: 文档已完成编写，正在进行内部评审
- **已批准**: 文档经过评审并获得批准，可以开始实施
- **已实施**: 文档对应的 API 已被实现
- **已完成**: 文档已完成所有流程，正式关闭
- **已更新**: 文档已更新，重新进入评审流程

## 6. API 文档编写规范

### 6.1 语言要求
- 使用清晰、简洁、无歧义的语言
- 避免使用模糊的词汇（如"大概"、"可能"、"应该"等）
- 使用主动语态，避免被动语态

### 6.2 格式要求
- 遵循本规范定义的文档结构
- 使用 Markdown 格式编写
- 表格用于展示结构化数据
- 列表用于展示多条相关信息
- 代码块用于展示示例代码，使用适当的语法高亮
- 使用适当的标题层级，保持文档结构清晰

### 6.3 接口命名规范
- 使用动词+名词的方式命名接口（如 `getUser`, `createAgent` 等）
- 使用 RESTful API 设计风格
- 路径参数使用冒号前缀（如 `/users/:id`）
- 查询参数使用 kebab-case（如 `?page-size=10`）

### 6.4 响应格式规范
- 成功响应应包含 `data` 字段，用于存放响应数据
- 分页响应应包含 `pagination` 字段，用于存放分页信息
- 错误响应应包含 `error` 字段，用于存放错误信息
- 错误信息应包含 `code`、`message` 和 `details` 字段

### 6.5 示例代码规范
- 提供多种语言的示例代码（如 curl、JavaScript、Python 等）
- 示例代码应完整、可运行
- 示例代码应包含必要的注释

## 7. API 文档管理流程

### 7.1 文档创建
1. 根据需求文档和设计文档创建 API 文档
2. 填写文档的详细信息，包括接口定义、参数、响应等
3. 提供示例代码和数据模型

### 7.2 文档评审
1. 文档完成后进行内部评审
2. 收集评审意见
3. 根据评审意见修改文档

### 7.3 文档批准
1. 评审通过后获得项目负责人批准
2. 文档正式生效

### 7.4 文档实施
1. 开发人员根据文档实现 API
2. 测试人员根据文档测试 API

### 7.5 文档更新
1. API 发生变更时，及时更新文档
2. 更新文档版本和变更日志
3. 重新进行评审和批准

## 8. 文档管理

### 8.1 版本控制
- 所有 API 文档应纳入版本控制
- 版本号遵循语义化版本规范（MAJOR.MINOR.PATCH）
- 每次更新应增加版本号并记录变更内容

### 8.2 存储位置
- 所有 API 文档应存储在 `docs/api/` 目录下
- 文档命名遵循 `[模块]-api-[版本].md` 格式，如 `gateway-api-v1.0.md`

### 8.3 评审与审批
- API 文档编写完成后应进行内部评审
- 评审通过后由项目负责人批准
- 批准后的文档方可作为开发依据
